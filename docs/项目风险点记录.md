# CBOND ON 项目风险点记录

记录时间：2026-06-02

本文档记录第一轮项目接管扫描中发现的风险点。当前仅做记录，不代表已经修复。

## P0 / 生产口径需立即确认

### 1. Live DB 写入日期与本地交易日期不一致

- 影响链路：实盘选券、DB 写入、下游读取 `o_0001`。
- 相关位置：`cbond_on/app/usecases/live_runtime.py`、`cbond_on/infra/live/db_writer.py`。
- 现象：
  - 本地 `trade_list.csv` 中 `trade_date`、`target_day`、`sell_day` 指向目标交易日。
  - DB 写入前会把 `trade_date` 改成 `prev_trade_day`。
- 风险：
  - 如果下游表语义不是“信号日前一交易日”，则会导致实盘信号日期错位。
  - Dashboard、本地结果、DB 结果可能在日期维度上解释不一致。
- 建议：
  - 优先确认 `quant_factor_dev.researcher_gswzif.o_0001.trade_date` 的业务语义。
  - 若应代表实际交易日，应改为写入 `target_day`。
  - 若应代表信号归属日，需要在配置、文档、Dashboard 展示中显式标注。

### 2. Benchmark 卖出窗口与交易/label/live 口径不完全一致

- 影响链路：回测报告、超额收益、实盘 Dashboard 绩效解释。
- 相关位置：`cbond_on/config/benchmark/benchmark_config.json5`、`cbond_on/infra/benchmark/service.py`、`cbond_on/config/data/label_config.json5`、`cbond_on/config/backtest/backtest_config.json5`、`cbond_on/config/live/live_config.json5`。
- 现象：
  - label/backtest/live 默认卖出窗口为 `twap_0930_0945`。
  - benchmark 配置中卖出窗口为 `twap_0930_0939`。
  - Dashboard 绩效里可以通过 UI 选择卖出窗口，并且走 DataHub daily_twap 口径；benchmark service 配置则可能走 NFS window data。
- 风险：
  - 策略收益和 benchmark 收益使用不同卖出口径时，超额收益会被窗口差异污染。
  - 研究报告、实盘看板、正式回测可能给出不一致的基准表现。
- 建议：
  - 明确 benchmark 是否必须使用 `09:30-09:39`。
  - 如果没有特殊原因，统一到 `twap_0930_0945`。
  - 在报告中输出 benchmark buy/sell 窗口，避免口径隐性变化。

## P1 / 运行稳定性与可维护性

### 3. Factor batch 在 benchmark 缺失分支可能触发未赋值变量

- 影响链路：单因子回测、factor screening、bad factor report。
- 相关位置：`cbond_on/app/usecases/factor_batch_runtime.py`。
- 现象：
  - 缺 benchmark 的分支里存在使用 `ic`、`rank_ic` 的路径。
  - 该分支可能发生在变量赋值之前。
- 风险：
  - 某些日期 benchmark 缺失时，批量因子回测可能直接异常中断。
- 建议：
  - 将 `ic/rank_ic` 的计算提前，或在缺 benchmark 分支显式写入 `NaN`。
  - 加一个 benchmark 缺失日期的单元测试或最小集成测试。

### 4. Scheduler STOP flag 不是调度循环的强制退出条件

- 影响链路：实盘调度、Dashboard 急停、运维控制。
- 相关位置：`liveLaunch/scheduler.py`、`liveLaunch/web/app.py`。
- 现象：
  - Web 层会写 STOP flag。
  - scheduler 主循环目前主要依赖进程 kill/stop 来停止。
- 风险：
  - 用户看到“停止标记”后，可能误以为调度器会主动检测并退出。
  - 极端情况下，状态标记和实际进程状态不一致。
- 建议：
  - 在 scheduler 循环中读取 STOP flag，并作为安全退出条件。
  - Dashboard 同时展示“标记状态”和“实际进程状态”。

### 5. Python 依赖缺少统一声明，当前环境 pytest 不可用

- 影响链路：本地复现、CI、部署、测试。
- 相关位置：项目根目录、`tests/`、运行环境。
- 现象：
  - 未发现清晰的 Python `requirements.txt`、`pyproject.toml` 或 `setup.py`。
  - 当前 Python 3.11 环境执行 `pytest` 时提示缺少 pytest。
- 风险：
  - 新机器或服务器复现成本高。
  - 守卫测试无法稳定纳入 CI。
- 建议：
  - 补充最小依赖清单。
  - 将 architecture guard、repo hygiene guard、核心 runtime smoke test 纳入固定验证流程。

### 6. Rust 因子 manifest 与实际 runtime registry 不一致

- 影响链路：因子覆盖检查、Rust 引擎升级、live 因子可用性判断。
- 相关位置：`rust/factor_engine/factor_manifest.json`、`cbond_on/domain/factors/defs/`、`cbond_on/infra/factors/rust_backend.py`。
- 现象：
  - 运行时 registry 当前可注册 99 个因子实现。
  - `factors_all_in_one` 当前有 122 个 factor specs。
  - Rust manifest 中的覆盖状态仍有陈旧/pending 信息，不能直接代表实际可运行覆盖。
- 风险：
  - 新增或排查因子时，如果依赖 manifest，会误判 Rust 覆盖状态。
- 建议：
  - 用运行时 registry + Rust 实际分支重新生成 manifest。
  - 增加 “factor_config -> registry -> rust backend” 的一致性检查。

### 7. Universe 配置字段与当前生产语义存在错位

- 影响链路：模型训练、回测、实盘选券。
- 相关位置：`cbond_on/infra/universe/pool_filter.py`、`cbond_on/infra/model/impl/lgbm/trainer.py`、`cbond_on/app/usecases/backtest_runtime.py`、`cbond_on/app/usecases/live_runtime.py`。
- 现象：
  - 配置中仍保留部分 `security_banlist`、流动性过滤、价格上下限等能力。
  - 当前生产 runtime 强制使用 `o_0005` allowlist，并禁止或忽略部分传统过滤字段。
- 风险：
  - 使用者看配置时容易误以为这些过滤仍然生效。
  - 研究、回测、实盘对 universe 的理解可能产生偏差。
- 建议：
  - 将生产配置中的无效字段清理或迁移到 legacy 配置。
  - 在配置注释中明确 “当前生产只认 o_0005 allowlist”。

## P2 / 口径透明度与长期治理

### 8. 配置 schema 较浅，关键约束主要藏在 runtime assert 中

- 影响链路：配置变更、上线前校验、排障。
- 相关位置：`cbond_on/config/schemas.py`、`cbond_on/bootstrap/`、`cbond_on/app/usecases/`。
- 现象：
  - schema 主要做浅层字段检查。
  - 真实生产约束，例如禁止 live raw/redis refresh、强制 o_0005、禁止 security_banlist，主要在 runtime 里 assert。
- 风险：
  - 配置问题可能到运行中后段才暴露。
  - 新人或自动化流程难以提前判断配置是否合法。
- 建议：
  - 补充 production/live/backtest 专用 config validator。
  - 将强约束前移到 bootstrap 阶段。

### 9. 旧链路代码仍保留，容易和当前生产链路混淆

- 影响链路：代码阅读、维护、排障。
- 相关位置：`cbond_on/infra/backtest/runner.py`、`cbond_on/infra/data/snapshot_loader.py`、`cbond_on/infra/live/redis_snapshot.py`、`cbond_on/config/data/raw_data_config.json5`、`cbond_on/infra/model/runners/lob/`。
- 现象：
  - 项目中仍存在旧 snapshot/TWAP、自建 raw/redis、旧 LOB dataset、旧 backtest runner 等能力。
  - 当前生产主链路并不依赖这些路径。
- 风险：
  - 排障时可能误读旧代码为生产路径。
  - 后续修改时可能改错链路。
- 建议：
  - 给 legacy 路径加明确注释或移动到 legacy namespace。
  - 在架构文档中维护“当前生产链路”和“历史保留链路”的边界。

### 10. Live 数据 ready gate 只校验 DataHub publish 状态，不校验字段级完整性

- 影响链路：实盘 panel/factor/model_score。
- 相关位置：`cbond_on/infra/live/publish_gate.py`、`cbond_on/app/usecases/live_runtime.py`。
- 现象：
  - ready gate 检查 manifest/done marker/status。
  - 后续字段完整性依赖 panel/factor/model runtime 自己报错。
- 风险：
  - DataHub 标记成功但缺字段、字段异常、日内窗口缺数据时，错误会在较晚阶段暴露。
- 建议：
  - 增加 live preflight，校验当日 clean panel 必要字段、daily_twap 必要列、o_0005 pool 是否存在。
  - 将 preflight 结果写入 Dashboard 状态。

## 已验证事项

- 架构边界守卫通过：`architecture guard: ok`。
- 仓库卫生守卫通过：`repo hygiene guard: ok`。
- 当前未执行完整 pytest，因为本地 Python 3.11 环境缺少 `pytest`。

## 建议后续修复顺序

1. 确认并修正 Live DB `trade_date` 口径。
2. 统一或显式标注 benchmark sell TWAP 窗口。
3. 修复 factor batch benchmark 缺失分支。
4. 补依赖声明和最小测试/守卫执行脚本。
5. 重建 Rust factor manifest 和一致性检查。
6. 清理或标注 legacy 链路。
