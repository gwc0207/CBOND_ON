# CBOND_ON 因子开发规则

本文档给自动开发因子的 Agent 使用。目标是让新因子能稳定进入 CBOND_ON 的研究、batch、模型和实盘链路，并且不引入未来数据、隐式兜底、字段猜测和 Rust/Python 不一致。

## 1. 总原则

新因子只能基于项目已经提供给因子引擎的上下文开发：

- `ctx.panel`：可转债 T1430 面板。
- `ctx.stock_panel`：正股 T1430 面板，仅在因子声明 `requires_stock_panel = True` 后加载。
- `ctx.bond_stock_map`：转债到正股映射，仅在因子声明 `requires_bond_stock_map = True` 后加载。
- `ctx.daily_data`：日频 DataHub 表，仅在因子通过 `daily_requirements()` 声明后加载。

新因子不允许直接读文件、连库、访问 Redis、读取 notebook、扫描 raw snapshot，也不允许自己实现数据处理兜底。数据准备、面板构建和日频表读取由现有 pipeline 完成，因子只消费上下文。

## 2. 时间口径

当前实盘因子口径：

- 面板名称：`T1430`
- 因子时间：`14:30`
- label 时间：`14:42`
- live 因子配置：`cbond_on/config/live/live_factors_config.json5`
- batch 因子配置：`cbond_on/config/factor/factor_config.json5`

时间可见性规则：

- 盘中因子只能使用 `14:30` 及之前已经进入 T1430 panel 的信息。
- 日频历史因子可以使用历史日频数据，但必须在因子内部通过 shift 或日期对齐保证不使用当日未来收益。
- `daily_twap` 可以用于历史隔夜收益类因子，但当前 T 日尚未完成的下午 TWAP 不得作为 T 日 14:30 因子输入。
- label、未来收益、回测结果、交易清单收益分析不能反向进入因子。

## 4. 禁止项

新因子不得做以下事情：

- 不得直接读取 `D:/cbond_data_hub`、`D:/cbond_on` 或任意本地文件。
- 不得连接数据库或 Redis。
- 不得读取 raw snapshot 或自己重建面板。
- 不得自己构造交易日历兜底。
- 不得在缺字段时静默返回 0。
- 不得用宽泛 `try/except` 吞掉错误。
- 不得用 `fillna(0)` 掩盖数据缺失，除非公式定义明确要求，并在因子说明里写清楚。
- 不得使用未来收益、label、交易清单收益、回测收益作为输入。
- 不得在 live profile 中加入没有 Rust kernel 的因子，除非 live 明确切回 Python 引擎。

## 5. Python 因子实现规范

### 5.1 文件位置

新因子实现放在：

```text
cbond_on/domain/factors/defs/<factor_key>.py
```

`factor_key` 使用小写 snake_case，并带版本号，例如：

```text
stock_bond_return_gap_v1
```

### 5.2 注册方式

必须用 `FactorRegistry` 注册：

```python
from cbond_on.core.registry import FactorRegistry
from cbond_on.domain.factors.base import Factor, FactorComputeContext


@FactorRegistry.register("stock_bond_return_gap_v1")
class StockBondReturnGapV1Factor(Factor):
    name = "stock_bond_return_gap_v1"

    def compute(self, ctx: FactorComputeContext):
        ...
```

同时必须在以下文件增加 import 和 `__all__`：

```text
cbond_on/domain/factors/defs/__init__.py
```

否则 CLI 启动时不会把因子注册进 `FactorRegistry`。

### 5.3 compute 返回值

`compute(ctx)` 必须返回 `pd.Series`。

返回索引必须是按 `(dt, code)` 对齐的结果。当前常见方式是从 panel 的 `(dt, code, seq)` 聚合到 `(dt, code)`：

```python
out.name = self.output_name(self.name)
return out
```

不得返回 DataFrame、list、numpy array 或无索引 Series。

### 5.4 参数规范

参数从 `ctx.params` 读取，并在配置中显式给出。

推荐写法：

```python
window = int(ctx.params.get("window", 20))
price_col = str(ctx.params.get("price_col", "last"))
```

参数必须有清楚默认值，但默认值不能改变数据口径。影响口径的参数必须写入 config。

### 5.5 字段检查

使用字段前必须检查字段存在：

```python
if price_col not in panel.columns:
    raise KeyError(f"factor_name missing column: {price_col}")
```

缺字段应失败并给出明确错误，不允许换字段兜底。

## 6. 上下文声明规则

### 6.1 使用正股 panel

需要正股数据时，在 Factor 类上声明：

```python
requires_stock_panel = True
requires_bond_stock_map = True
```

pipeline 会加载：

- `ctx.stock_panel`
- `ctx.bond_stock_map`

没有这两个声明时，因子不得访问正股数据。

### 6.2 使用日频表

需要日频数据时，必须实现 `daily_requirements()`：

```python
from cbond_on.domain.factors.base import DailyFactorRequirement

@classmethod
def daily_requirements(cls, params: dict | None = None) -> list[DailyFactorRequirement]:
    params = dict(params or {})
    window = int(params.get("window", 20))
    return [
        DailyFactorRequirement(
            source="market_cbond.daily_twap",
            columns=("twap_1442_1457", "twap_0930_1000"),
            lookback_days=window + 2,
        )
    ]
```

`compute()` 中只能从 `ctx.daily_data[source]` 取数：

```python
daily = ctx.daily_data.get("market_cbond.daily_twap")
```

不得在 `compute()` 里调用文件读取函数。

## 7. OHLC 重构规则

当前 panel 是快照序列，部分因子会在 Python/Rust 内部基于回看窗口重构局部 OHLC 或 bar 特征。

开发要求：

- 使用重构 OHLC 时必须说明窗口长度，例如 `windowsize=10`。
- 只能基于 `ctx.panel` 或 `ctx.stock_panel` 内已有快照重构。
- 不允许读取分钟线文件作为额外输入。
- 不允许在因子内重新定义 panel 回看范围；回看范围由 `panel_config.json5` 控制。
- 如果新增多个 `windowsize`，要注意 Rust planning 限制：当前唯一窗口数过多会 fail-fast。

## 8. Rust 配合规则

当前 live 因子配置中：

```text
compute.engine = "rust"
```

因此任何进入 live 的新因子都必须满足以下条件之一：

- 已在 Rust 中实现 kernel；
- 或者 live 配置明确不使用 Rust。

当前 Rust 行为：

- Python 通过 `cbond_on/infra/factors/rust_backend.py` 调用 `cbond_on_rust.compute_factor_frame`。
- 输入包括 `panel_df, specs_payload, stock_df, map_df, daily_data, compute_backend_params`。
- 输出必须是 DataFrame，且包含 `dt`, `code` 和每个 factor spec 对应的输出列。
- Rust 模式是 fail-fast；没有 kernel 会直接失败，不允许静默回退 Python。

新增 Rust kernel 时至少要做：

1. 在 `rust/factor_engine/src/lib.rs` 增加对应 factor 分支和计算实现。
2. 确保 Rust 输出列名等于 `spec.output_col or spec.name`。
3. 如果使用日频数据，读取 Rust 收到的 `daily_data[source]`，字段名必须和 Python `daily_requirements()` 一致。
4. 更新 `rust/factor_engine/factor_manifest.json` 中该因子的 `rust_status`。
5. 重建本地扩展，参考 `rust/factor_engine/README.md`。

Windows 重建命令摘要：


## 9. 注册和入库管理

一个新因子要进入项目，至少需要完成四层注册。

### 9.1 Python 注册

修改：

```text
cbond_on/domain/factors/defs/<factor_key>.py
cbond_on/domain/factors/defs/__init__.py
```

确认：

- `@FactorRegistry.register("<factor_key>")` 存在。
- `__init__.py` import 了新类。
- `__all__` 包含新类名。

### 9.2 因子合同注册

当前合同目录：

```text
cbond_on/factor_contracts/registry.json5
cbond_on/factor_contracts/profiles/live.json5
cbond_on/factor_contracts/profiles/model_score.json5
cbond_on/factor_contracts/profiles/research.json5
```

新增因子必须在 `registry.json5` 增加记录，至少说明：

```text
name
implementation
family
uses_ohlc_rebuild
live_enabled
model_enabled
```

如果进入 live，则还要加入：

```text
cbond_on/factor_contracts/profiles/live.json5
```

### 9.3 配置注册

研究 batch 使用：

```text
cbond_on/config/factor/factor_config.json5
cbond_on/config/factor/*.json5
```

实盘使用：

```text
cbond_on/config/live/live_factors_config.json5
```

配置项格式：

```json5
{ name: "factor_instance_name", factor: "factor_key", params: { window: 20 } }
```

其中：

- `factor` 是 `FactorRegistry` 的实现名。
- `name` 是输出列名和模型特征名。
- 同一个 `factor` 可以用不同 `name + params` 生成多个实例。

### 9.4 模型特征注册

如果因子进入模型，需要同步检查模型配置和 feature list。当前 live 模型由：

```text
cbond_on/config/live/live_config.json5
cbond_on/config/live/live_models_config.json5
cbond_on/config/models/
```

中的 `model_id` 和模型配置决定。

原则：

- 因子只进 batch，不代表进模型。
- 因子只进模型，不代表进 live。
- 进入 live 前必须确认 factor 文件、model feature、Rust kernel、factor_contracts profile 一致。

### 10.2 live 因子构建

live 使用：

```text

```

调度链路会按 live config 构建 panel、labels、factors，然后模型打分、策略选券、生成交易清单。

手动把新因子加入 live 前必须确认：

- batch 单日能生成。
- batch 多日能生成。
- 缺失率、inf、异常值已检查。
- Rust kernel 已覆盖。
- live model 的 feature list 已匹配。
- factor_contracts live profile 已更新。

## 11. 质量检查

每个新因子至少要通过以下检查：

- import 检查：`cbond_on.domain.factors.defs` 能正常导入。
- 注册检查：`FactorRegistry.get("<factor_key>")` 能取到类。
- 单日计算检查：指定交易日能产出非空结果。
- 多日 batch 检查：覆盖目标日期区间。
- 缺失率检查：说明 NaN 来源，不允许默认填 0 掩盖。
- 极值检查：检查 inf、-inf 和异常大值。
- 横截面检查：每天有效样本数足够。
- Rust/Python 对齐：如进 live，Rust 输出必须和 Python 输出同口径。
- 时间口径检查：不使用未来数据。
- live 准入检查：factor_config、live_factors_config、factor_contracts、model feature、Rust manifest 一致。

## 13. 最小示例

```python
from __future__ import annotations

import pandas as pd

from cbond_on.core.registry import FactorRegistry
from cbond_on.domain.factors.base import Factor, FactorComputeContext
from cbond_on.domain.factors.defs._intraday_utils import ensure_trade_time, _group_scalar, slice_window


@FactorRegistry.register("example_last_return_v1")
class ExampleLastReturnV1Factor(Factor):
    name = "example_last_return_v1"

    def compute(self, ctx: FactorComputeContext) -> pd.Series:
        panel = ensure_trade_time(ctx.panel)
        window_minutes = int(ctx.params.get("window_minutes", 30))
        price_col = str(ctx.params.get("price_col", "last"))
        if price_col not in panel.columns:
            raise KeyError(f"example_last_return_v1 missing column: {price_col}")

        def _calc(df: pd.DataFrame) -> float:
            df = slice_window(df.sort_values("trade_time"), window_minutes)
            px = pd.to_numeric(df[price_col], errors="coerce").dropna()
            if len(px) < 2:
                return float("nan")
            first = float(px.iloc[0])
            last = float(px.iloc[-1])
            if first <= 0:
                return float("nan")
            return last / first - 1.0

        out = _group_scalar(panel, _calc)
        out.name = self.output_name(self.name)
        return out
```

这个示例只展示形态，不代表推荐因子。正式因子必须补充合同、配置、验证和 Rust 状态。
